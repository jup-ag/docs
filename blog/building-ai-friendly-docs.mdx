We built docs that developers loved. Then we watched AI agents try to use them,
and learned that good and navigable are not the same thing.

Your documentation has a new first reader. When developers integrate a new
protocol today, their first move is increasingly to hand the docs to an AI
agent and say "figure it out." The AI reads your docs, interprets them, and
writes code the developer ships. If it gets it right, the developer has a
working integration in minutes. If it gets it wrong, they get confidently
broken code — and may not catch it until something fails in production.

This is the multiplier effect. Good docs have always mattered, but now they
compound. Every improvement in structure and clarity is amplified across every
AI tool that reads your pages. Every gap and ambiguity is amplified too.

This post explores how we responded to that shift — with two principles:
making docs the single source of truth for both humans and AI, and meeting
AI tools wherever they already work.

---

## Your Docs Have a New First Reader

The immediate instinct is to bolt something on. Publish an `llms.txt`. Train
a chatbot. Write "machine-optimised" descriptions. Each helps in isolation,
but they don't address the deeper shift: AI is no longer a secondary consumer
of your documentation. It's the primary one. And the way it reads, and fails,
is fundamentally different from humans.

A human reads a confusing paragraph and asks for clarification. An AI reads
it and generates code that looks right but isn't. The failure mode isn't
confusion or misunderstanding, it's confidently wrong. That raises the stakes
on everything downstream: what your AI reads, how it reads it, and whether
what it reads is current.

When we started building AI support into our docs, two things sharpened this for us.

**AI tools don't all read the same way.** This isn't a theoretical concern —
you can see it in how the standards have emerged independently.
[llmstxt.org](https://llmstxt.org) defines a structured index for LLM
discovery. [Model Context Protocol](https://modelcontextprotocol.io/) defines
a live query interface for AI editors. [agentskills.io](https://agentskills.io)
defines action schemas for agent frameworks. These standards exist because
they solve different problems: a RAG pipeline wants to ingest your entire site
as a single file, an MCP client wants to query specific pages in real time,
an agent framework wants structured action definitions with input/output
schemas, and a build script just wants to curl one page as raw markdown.
Optimising for one doesn't help the others. It is not a one size fit all
solution, instead, there's a landscape of consumption patterns, and it's
going to keep evolving.

**What AI reads needs to stay current.** This is a well-known problem in
software engineering — it's why OpenAPI generates from code, why JSDoc
generates from source, why every team that's maintained parallel
documentation has eventually watched one version go stale. The same principle
applies here. The moment you maintain an "AI version" of your docs or resource 
alongside the human version, you've created a synchronisation problem. Content 
diverges. Specs go stale. The LLM index says one thing; the actual endpoint does 
another. We haven't maintained parallel AI resources long enough to feel this
pain yet, but the pattern is predictable. And when AI is the first reader,
stale content doesn't just confuse — it generates wrong code confidently at 
scale.

So the question we kept coming back to: if AI is the first reader, how do you
make sure it reads well — across every tool, every workflow, every time?

---

## One Source, Many Outputs

The approach we took: make the documentation itself the single source from
which every AI entry point is derived. No separation of content. No parallel
maintenance. Enrich the existing source, then generate the paths.

It starts with the API layer. Jupiter's APIs are developer experience first — 
API-first, no RPC required, no mandatory API keys, clean JSON in and JSON out. 
This isn't an AI-specific design decision, but it's the foundation that makes 
everything else possible. Any agent that can make HTTP calls can interact with 
Jupiter. No SDK installation, no blockchain node setup, no binary dependencies.

With that foundation in place, the next layer is the documentation itself.
The goal: enrich existing pages with just enough machine-readable metadata
that every AI consumption path can be generated from the same source.

#### llms.txt
[`llms.txt`](https://dev.jup.ag/llms.txt) is auto-generated from page
frontmatter and navigation config, following the
[llmstxt.org](https://llmstxt.org) standard. It's a structured Markdown
index of every API, guide, and reference page with its `llmsDescription`. For
RAG pipelines that need complete content,
[`llms-full.txt`](https://dev.jup.ag/llms-full.txt) concatenates the entire
site into a single file.

The `llmsDescription` is a new frontmatter field, we have added to every page 
alongside the existing human-facing `description`, the change is small yet effective.

```yaml
---
title: "Ultra Swap API"
description: "Overview of Ultra Swap and its features."
llmsDescription: "Jupiter Ultra Swap API provides a managed swap execution
  engine. POST to /ultra/v1/order for quotes, /ultra/v1/execute to submit.
  Handles routing, slippage, gas, MEV protection server-side. No RPC or
  wallet infrastructure required."
---
```

- `description` is short and scannable: written for a human browsing the site.
- `llmsDescription` is descriptive and detailed, more technical: specific endpoints,
key capabilities, what the API actually does and how to call it. Both live on the
same page. One update to the source, and both descriptions are current.

#### skill.md
Most agent "skills" or "tools" in the ecosystem are individual function
definitions — one tool per API call. Give an agent 50 endpoint definitions
and it has to figure out which one matches the developer's intent. That means
loading every tool into context, parsing each schema, and hoping the LLM
picks the right one. It works for simple integrations, but a protocol with
dozens of endpoints across multiple products needs something more intentional.

[`skill.md`](https://dev.jup.ag/skill.md) follows the
[agentskills.io](https://agentskills.io) specification, but takes a different
approach: intent routing. Instead of one tool per endpoint, it organises
Jupiter's capabilities by what a developer actually wants to do. An agent
that receives "swap tokens" gets routed to the Ultra API family and its first
action. "Check price" gets routed to the Price API. The agent doesn't need to
know Jupiter's full API surface — it expresses intent, and the skill handles
the mapping.

This is more robust than a flat list of tools. If endpoints are added or APIs
are restructured, the intent categories stay stable. It's also more
token-efficient: instead of 50 tool schemas in context, the agent loads one
skill definition with intent categories and lets the router resolve the rest.

`skill.md` is sourced directly from the open-source
[Skills Repository](https://github.com/jup-ag/agent-skills), which also
ships framework-specific definitions ready to drop in:

- **Claude / OpenAI:** function calling with tool definitions
- **Vercel AI SDK:** Zod-typed tool schemas
- **LangChain:** `@tool` decorated functions

#### MCP
Most AI tools today get their context from web searches. The problem is that
search results are cached and indexed — not live. When an API changes, search
engines may not reflect it for days or weeks. AI assistants confidently
generate code against outdated endpoints, deprecated parameters, and stale
examples. The developer gets working-looking code that breaks against the
current API.

The [MCP server](https://dev.jup.ag/mcp) sidesteps this entirely. Via the
[Model Context Protocol](https://modelcontextprotocol.io/), AI editors query
the documentation source directly — the same source that generates `llms.txt`
and `skill.md`. No caching layer. No indexing delay. The context your AI
reads is always the context that's currently live.

#### Markdown Export
Markdown export renders the same pages in a different format. Append `.md`
to any documentation URL or set the `Accept: text/markdown` header.

#### OpenAPI Specs
OpenAPI specs live in the docs repo and feed both human-readable API
references and machine-readable schemas.

Every path reads from the same source. Update the source or page's frontmatter, and the
change propagates through `llms.txt`, gets picked up by MCP queries, flows
into skill definitions, and reflects in any tool that processes the page. One
input, many outputs.

## Wherever Your AI Already Works

This is where the two principles converge. AI tooling is fragmented — not
because the ecosystem can't agree on a standard, but because they are  different 
tools solving different problems. Each one is its own entrypoint into our docs.

Because everything is derived from one source, meeting each of these doesn't mean 
maintaining separate systems — it means adding another output from the same source.

| How AI consumes docs | Jupiter resource |
|:---|:---|
| **High-level discovery** | `llms.txt` — structured index of all docs |
| **Full-context ingestion** | `llms-full.txt` — complete site content |
| **In-editor queries** | MCP server at `dev.jup.ag/mcp` |
| **Agent capabilities** | `skill.md` + [Skills Repository](https://github.com/jup-ag/agent-skills) |
| **Single page fetch** | `.md` suffix on any URL |
| **API schema** | OpenAPI specs for every product |
| **LLMs friendly description** | `llmsDescription` frontmatter on every page |

The principle: don't force AI into one consumption pattern. Meet it wherever
it already works.

The landscape is ever shifting. New standards will emerge, new consumption
patterns will appear, and the way AI tools read documentation will keep
evolving. We will continue to ensure adaptability, and keep our docs 
the single source of truth for both humans and AI, and meeting AI tools wherever 
they already work.

If you're building on Jupiter with AI, or building AI-friendly docs for your
own project, we'd love to hear what's working and what isn't. Reach out on us.
