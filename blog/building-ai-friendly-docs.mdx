We built docs that developers loved. Then we watched AI agents try to use them, and learned that good and navigable are not the same thing.

Developers are already building on Jupiter with AI. Claude Code is scaffolding integrations. Cursor is autocompleting API calls. Autonomous agents are executing swaps. AI tools are reading our docs before developers do — but they don't browse the way humans do. They don't scan headings or click through navigation. They need structured paths to the right information.

And there's no single path that works for all of them. RAG pipelines need bulk content. MCP clients need live queries. Agent frameworks need structured action definitions. Scripts need raw markdown. So we built for all of them — multiple entry points, all derived from a single source, on top of APIs designed for simplicity from the start: no RPC nodes, no mandatory API keys, clean REST with JSON in and JSON out.

Here's how it comes together, layer by layer.

**A few terms you'll see throughout this post:**

- **LLM (Large Language Model):** the AI models powering tools like ChatGPT, Claude, and Cursor
- **RAG (Retrieval-Augmented Generation):** a pattern where AI tools fetch relevant documents before generating a response, improving accuracy
- **MCP (Model Context Protocol):** an open protocol that lets AI editors query external data sources directly

---

## Give Your Agent a Starting Point

The first thing an AI agent needs is a way in — not the full docs, just enough to know what exists and where to find it.

Jupiter's [`llms.txt`](https://dev.jup.ag/llms.txt) follows the [llmstxt.org](https://llmstxt.org) standard: a structured Markdown file with a curated index of every API, guide, and reference page, each with a title and description. Your agent reads one file and understands the full landscape — what Jupiter offers, where to find it, and what each page covers.

For teams building RAG pipelines or needing complete context, there's [`llms-full.txt`](https://dev.jup.ag/llms-full.txt): the entire documentation site as a single file.

Both are auto-generated from our docs structure. Every time a page ships, the index updates. No manual curation.

Your agent goes from "I need to integrate Jupiter" to "here's the exact endpoint" in one file read.

## Write for Both Readers

A starting point gets your agent to the right page. What it finds there matters just as much.

Every page in our docs carries two descriptions in its frontmatter:

```yaml
---
title: "Ultra Swap API"
description: "Overview of Ultra Swap and its features."
llmsDescription: "Jupiter Ultra Swap API provides a managed swap execution
  engine. POST to /ultra/v1/order for quotes, /ultra/v1/execute to submit.
  Handles routing, slippage, gas, MEV protection server-side. No RPC or
  wallet infrastructure required."
---
```

`description` is for humans browsing the site — short, scannable, written for context. `llmsDescription` is for AI — more descriptive and technical, with specific endpoints, key capabilities, and outcomes.

When an AI tool reads `llms.txt` or processes page frontmatter, it gets a description optimised for machine consumption. Richer context for your AI, without cluttering the human experience.

## Bring Docs into the Workflow

Good context on the page is one thing. Getting it to where developers actually work is another.

**In-editor access via MCP**

The [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) lets AI editors query external data sources in-context. Jupiter's docs expose a native MCP server at:

```bash
https://dev.jup.ag/mcp
```

Connect it to Claude Code, Cursor, Windsurf, or any MCP-compatible tool. Your AI assistant gets direct access to all Jupiter documentation, OpenAPI specs, and code examples without leaving the editor.

This is more effective than web searching — MCP provides the direct source, so your AI isn't working from cached or outdated content. Your assistant can look up endpoints, read parameter schemas, scaffold API calls, and debug errors against the real specification.

**Single-page fetch**

Sometimes an agent just needs one specific page. Any Jupiter documentation URL is accessible as raw markdown:

```bash
curl https://dev.jup.ag/docs/ultra.md
```

Or via the `Accept` header:

```bash
curl -H "Accept: text/markdown" https://dev.jup.ag/docs/ultra
```

Clean markdown, ready to parse. No HTML stripping required.

Between `llms.txt` for discovery, MCP for in-editor access, and `.md` for direct fetch, your agents have multiple paths to the same information — depending on how they work.

## From Reading to Doing

Everything so far helps AI tools find and understand documentation. But developers want their agents to act, not just read.

That's the difference between `llms.txt` and [`skill.md`](https://dev.jup.ag/skill.md). Where `llms.txt` helps an agent discover what's available, `skill.md` — following the [agentskills.io](https://agentskills.io) specification — tells it exactly how to use it: input/output schemas, API endpoints, trigger keywords, and example payloads. An agent framework reads this file and knows how to interact with Jupiter without custom integration code.

We also open-sourced the [Jupiter Skills Repository](https://github.com/jup-ag/agent-skills) with ready-to-use definitions for popular frameworks:

- **Claude / OpenAI:** function calling with tool definitions
- **Vercel AI SDK:** Zod-typed tool schemas
- **LangChain:** `@tool` decorated functions

The repository includes an intent router that maps developer intent (like "swap tokens" or "check price") to the right API family and first action. Define the skill once, and any LLM with function calling can execute swaps, fetch prices, and manage positions.

## One Source, No Drift

None of this matters if it falls out of sync. That's the architectural decision underneath everything: every AI entry point is derived from or served directly from the docs repo. There's no separate "AI version" to maintain.

Update a page's frontmatter, and the change flows through:

- `llms.txt` regenerates from frontmatter and navigation config
- `skill.md` regenerates from docs structure and OpenAPI specs
- MCP serves the updated content directly
- Markdown export renders the same page in a different format
- OpenAPI specs feed both human-readable references and machine-readable schemas

One input, many outputs. One update, every path reflects it.

| How AI consumes docs | Jupiter resource |
|:---|:---|
| **High-level discovery** | `llms.txt` — structured index of all docs |
| **Full-context RAG** | `llms-full.txt` — complete site content |
| **In-editor queries** | MCP server at `dev.jup.ag/mcp` |
| **Agent capabilities** | `skill.md` + [Skills Repository](https://github.com/jup-ag/agent-skills) |
| **Single page fetch** | `.md` suffix on any URL |
| **API schema** | OpenAPI specs for every product |
| **Dual descriptions** | `llmsDescription` frontmatter on every page |

The principle: don't force AI into one consumption pattern. Meet it wherever it already works.
