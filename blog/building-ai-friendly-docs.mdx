We built docs that developers loved. Then we watched AI agents try to use them, and learned that good and navigable are not the same thing.

Your documentation has a new first reader. When developers integrate a new protocol today, their first move is increasingly to hand the docs to an AI agent and say "figure it out." The AI reads your docs, interprets them, and writes code the developer ships. If it gets it right, the developer has a working integration in minutes. If it gets it wrong, they get confidently broken code — and may not catch it until something fails in production.

This is the multiplier effect. Good docs have always mattered, but now they compound. Every improvement in structure and clarity is amplified across every AI tool that reads your pages. Every gap and ambiguity is amplified too.

This post explores how Jupiter responded to that shift — with two principles: making docs the single source of truth for both humans and AI, and meeting AI tools wherever they already work.

**A few terms used in this post:**

- **LLM (Large Language Model):** the AI models powering tools like ChatGPT, Claude, and Cursor
- **RAG (Retrieval-Augmented Generation):** a pattern where AI tools fetch relevant documents before generating a response, improving accuracy
- **MCP (Model Context Protocol):** an open protocol that lets AI editors query external data sources directly

## Your Docs Have a New First Reader

The immediate instinct is to bolt something on. Publish an `llms.txt`. Train a chatbot. Write "machine-optimised" descriptions. Each helps in isolation, but they don't address the deeper shift: AI is no longer a secondary consumer of your documentation. It's the primary one. And the way it reads — and fails — is fundamentally different from humans.

A human reads a confusing paragraph and asks for clarification. An AI reads it and generates code that looks right but isn't. The failure mode isn't confusion — it's confidence. That raises the stakes on everything downstream: what your AI reads, how it reads it, and whether what it reads is current.

When we started building AI support into our docs, two things sharpened this for us.

**AI tools don't all read the same way.** This isn't a theoretical concern — you can see it in how the standards have emerged independently. [llmstxt.org](https://llmstxt.org) defines a structured index for LLM discovery. [Model Context Protocol](https://modelcontextprotocol.io/) defines a live query interface for AI editors. [agentskills.io](https://agentskills.io) defines action schemas for agent frameworks. These standards exist because they solve different problems: a RAG pipeline wants to ingest your entire site as a single file, an MCP client wants to query specific pages in real time, an agent framework wants structured action definitions with input/output schemas, and a build script just wants to curl one page as raw markdown. Optimising for one doesn't help the others. There's no single "AI-friendly" format — there's a landscape of consumption patterns, and it's still evolving.

**What AI reads needs to stay current.** This is a well-known problem in software engineering — it's why OpenAPI generates from code, why JSDoc generates from source, why every team that's maintained parallel documentation has eventually watched one version go stale. The same principle applies here. The moment you maintain an "AI version" of your docs alongside the human version, you've created a synchronisation problem. Content diverges. Specs go stale. The LLM index says one thing; the actual endpoint does another. Most teams haven't maintained parallel AI resources long enough to feel this pain yet, but the pattern is predictable. And when AI is the first reader, stale content doesn't just confuse — it generates wrong code at scale.

So the question we kept coming back to: if AI is the first reader, how do you make sure it reads well — across every tool, every workflow, every time?

## One Source, Many Outputs

The approach we took: make the documentation itself the single source from which every AI entry point is derived. No separate "AI docs." No parallel maintenance. Enrich the existing source, then generate the paths.

It starts with the API layer. Jupiter's APIs are REST-first — no RPC nodes, no mandatory API keys, clean JSON in and JSON out. This isn't an AI-specific design decision, but it's the foundation that makes everything else possible. Any agent that can make HTTP calls can interact with Jupiter. No SDK installation, no blockchain node setup, no binary dependencies.

On the docs layer, the enrichment is small: a single frontmatter field called `llmsDescription`, added to every page alongside the existing human-facing `description`.

```yaml
---
title: "Ultra Swap API"
description: "Overview of Ultra Swap and its features."
llmsDescription: "Jupiter Ultra Swap API provides a managed swap execution
  engine. POST to /ultra/v1/order for quotes, /ultra/v1/execute to submit.
  Handles routing, slippage, gas, MEV protection server-side. No RPC or
  wallet infrastructure required."
---
```

`description` is short and scannable — written for a human browsing the site. `llmsDescription` is longer, more technical — specific endpoints, key capabilities, what the API actually does and how to call it. Both live on the same page. One update to the source, and both descriptions are current.

From this enriched source, everything else is derived:

[`llms.txt`](https://dev.jup.ag/llms.txt) is auto-generated from page frontmatter and navigation config, following the [llmstxt.org](https://llmstxt.org) standard. It's a structured Markdown index of every API, guide, and reference page with its `llmsDescription`. For RAG pipelines that need complete content, [`llms-full.txt`](https://dev.jup.ag/llms-full.txt) concatenates the entire site into a single file.

[`skill.md`](https://dev.jup.ag/skill.md) is auto-generated from the documentation structure and OpenAPI specs, following the [agentskills.io](https://agentskills.io) specification. Where `llms.txt` tells an agent what's available, `skill.md` tells it exactly how to act — input/output schemas, API endpoints, trigger keywords, and example payloads.

The [MCP server](https://dev.jup.ag/mcp) serves documentation content directly via the [Model Context Protocol](https://modelcontextprotocol.io/). AI editors query it in real time — no crawling, no caching, no stale results.

Markdown export renders the same pages in a different format. Append `.md` to any documentation URL or set the `Accept: text/markdown` header.

OpenAPI specs live in the docs repo and feed both human-readable API references and machine-readable schemas.

Every path reads from the same source. Update a page's frontmatter, and the change propagates through `llms.txt`, gets picked up by MCP queries, flows into skill definitions, and reflects in any tool that processes the page. One input, many outputs. No drift.

## Wherever Your AI Already Works

This is where the two principles converge. Because everything is derived from one source, supporting a new AI consumption pattern doesn't mean maintaining a new system — it means adding another output.

| How AI consumes docs | Jupiter resource |
|:---|:---|
| **High-level discovery** | `llms.txt` — structured index of all docs |
| **Full-context ingestion** | `llms-full.txt` — complete site content |
| **In-editor queries** | MCP server at `dev.jup.ag/mcp` |
| **Agent capabilities** | `skill.md` + [Skills Repository](https://github.com/jup-ag/agent-skills) |
| **Single page fetch** | `.md` suffix on any URL |
| **API schema** | OpenAPI specs for every product |
| **Dual descriptions** | `llmsDescription` frontmatter on every page |

MCP deserves a note here. It's more effective than web searching because it serves the authoritative source directly — your AI assistant isn't synthesising from cached, possibly outdated search results. It's reading the actual specification. For developers building in AI editors, this means Jupiter context is always current and always available without leaving the editor.

The [Skills Repository](https://github.com/jup-ag/agent-skills) extends `skill.md` with framework-specific definitions:

- **Claude / OpenAI:** function calling with tool definitions
- **Vercel AI SDK:** Zod-typed tool schemas
- **LangChain:** `@tool` decorated functions

It includes an intent router that maps developer intent (like "swap tokens" or "check price") to the right API family and first action. Define the skill once, and any LLM with function calling can execute swaps, fetch prices, and manage positions.

The principle: don't force AI into one consumption pattern. Meet it wherever it already works.
