Advanced patterns for both the `/order` and `/build` paths.

## Compute unit estimation (`/build`)

The `/build` response includes `computeBudgetInstructions` with a default compute unit limit. For production use, simulate the transaction first to get an accurate CU estimate, then set a tighter limit to save fees.

```typescript
import { getSetComputeUnitLimitInstruction } from "@solana-program/compute-budget";

// 1. Build the transaction with all instructions (see /build guide)
// 2. Simulate to get actual CU used
const simulation = await rpc
  .simulateTransaction(getBase64EncodedWireTransaction(transaction), {
    encoding: "base64",
    commitment: "processed",
  })
  .send();

const unitsConsumed = simulation.value.unitsConsumed;

// 3. Replace the compute budget instruction with a tighter limit
// Add a 10-20% buffer for safety
const cuLimitIx = getSetComputeUnitLimitInstruction({
  units: Math.ceil(unitsConsumed * 1.15),
});
```

Replace the first instruction in `computeBudgetInstructions` with your simulated CU limit. Keep the compute unit price instruction as-is (or set your own).

## Removing no-op createIdempotent instructions

The `/build` response always includes `createAssociatedTokenAccountIdempotent` instructions in `setupInstructions`, even when the token accounts already exist. These are no-ops on existing accounts but still consume space in the transaction.

To reclaim that space for your own instructions:

1. Check onchain if the associated token accounts already exist
2. If they exist, omit those setup instructions from your transaction
3. If they don't exist, keep the instructions

```typescript
// Check if ATA exists before including the setup instruction
const ataAddress = await findAssociatedTokenPda({
  mint: address(outputMint),
  owner: address(userPublicKey),
  tokenProgram: TOKEN_PROGRAM_ADDRESS,
});

const accountInfo = await rpc.getAccountInfo(ataAddress[0]).send();

// Only include setup instruction if account does not exist
const filteredSetupInstructions = build.setupInstructions.filter((ix) => {
  // Compare with the createIdempotent instruction's target account
  // Keep instructions for accounts that don't exist yet
  return !accountExists(ix, existingAccounts);
});
```

## Fast mode (`/build`)

Set `mode=fast` on `/build` to reduce routing latency at the cost of quote optimality. This uses a Bellman-Ford algorithm with stricter constraints (fewer hops, fewer splits).

```
GET /build?inputMint=...&outputMint=...&amount=...&userPublicKey=...&mode=fast
```

Fast mode is useful when:
- You are swapping high-frequency and need low latency
- The pair is well-known (e.g. SOL/USDC) and doesn't benefit from complex routing
- You are building a trading bot where speed matters more than extracting the last basis point

## maxAccounts (`/build`)

Control transaction size with `maxAccounts` (1-64, default 64). Lower values produce simpler routes with fewer accounts, which is useful when:

- Your transaction already has many accounts from custom instructions
- You are hitting the 1232-byte transaction size limit
- You need room for CPI accounts

```
GET /build?...&maxAccounts=20
```

Reducing `maxAccounts` may result in worse pricing because the router has fewer DEXes and splits available.

## Gasless swaps (`/order`)

Enable gasless swaps by setting a `payer` on `/order`. The payer account covers gas-related fees (signature, priority, rent) on behalf of the taker.

```typescript
const params = new URLSearchParams({
  inputMint: "EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v", // USDC
  outputMint: "So11111111111111111111111111111111111111112",     // SOL
  amount: "1000000",
  taker: userWallet,
  payer: payerWallet, // Integrator's payer account
});
```

When `payer` is set and the taker lacks sufficient SOL for gas:
- Jupiter routes through a dedicated gasless flow (Mantis)
- The payer signs alongside the taker
- A platform fee is added to recoup gas costs

<Note>
  Setting manual slippage or priority fees alongside `payer` disables the gasless retry mechanism.
</Note>

Gasless is only available on `/order`. The `/build` path does not support gasless because you manage execution yourself.

## Priority fees (`/order`)

Control priority fees on `/order` with these parameters:

| Parameter | Description |
| --- | --- |
| `priorityFeeLamports` | Priority fee in lamports |
| `jitoTipLamports` | Jito MEV tip for faster inclusion |
| `broadcastFeeType` | `maxCap` (fee is a maximum) or `exactFee` (exact amount) |

Without these parameters, Jupiter automatically sets an appropriate priority fee. Setting them manually gives you control but overrides the automatic optimisation.

## Circular arbitrage routes

For circular arbitrage routing (e.g. SOL → USDC → SOL), use the [Metis binary](https://metis.builders). The Swap API V2 does not support circular routes.
