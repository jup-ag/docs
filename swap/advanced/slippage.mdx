Slippage is the difference between the quoted output amount and what you actually receive. It depends on token pair volatility, trade size relative to pool liquidity, and time between quoting and execution.

Jupiter's **Real-Time Slippage Estimator (RTSE)** dynamically calculates the optimal slippage tolerance for each swap, balancing trade success against price protection.

## How RTSE works

RTSE estimates slippage using three inputs:

- **Heuristics:** token categories (stablecoin, blue-chip, long-tail), historical slippage data, and real-time market conditions
- **Algorithms:** Exponential Moving Average (EMA) on observed slippage data to smooth out noise and track trends
- **Monitoring:** real-time failure rate tracking to reactively increase slippage when market conditions deteriorate

The result is a per-swap slippage value that is tight enough to protect against MEV and price impact, but loose enough for the transaction to land.

## RTSE on `/order` vs `/build`

| | `/order` | `/build` |
|---|---|---|
| **RTSE** | Automatic. Applied at execution time by `/execute`. | Opt-in. Pass `slippageBps=rtse`. |
| **Default** | RTSE (no param needed) | `slippageBps=50` (fixed 0.5%) |
| **Override** | Pass `slippageBps` for a fixed value (restricts to Metis-only routing) | Pass a number for fixed, or `"rtse"` for dynamic |

### /order (automatic)

On the `/order` + `/execute` path, RTSE is applied automatically at execution time. You do not need to set any slippage parameter. Jupiter adjusts slippage dynamically when the transaction is executed via `/execute`, using real-time market data at that moment.

If you pass `slippageBps` to `/order`, it overrides RTSE with a fixed value. This also restricts routing to Metis-only (see [Routing](/swap/routing)).

### /build (opt-in)

On the `/build` path, slippage defaults to 50 bps (0.5%). To enable RTSE, pass the literal string `"rtse"`:

```
GET https://api.jup.ag/swap/v2/build?inputMint=...&outputMint=...&amount=...&taker=...&slippageBps=rtse
```

When `slippageBps=rtse` is set, the API simulates the transaction and estimates optimal slippage using the same RTSE engine that powers `/order`. The estimated slippage is reflected in `otherAmountThreshold` in the response.

## When to use RTSE

**Use RTSE (recommended for most cases):**
- You want Jupiter to handle slippage optimisation for you
- You are swapping volatile or long-tail tokens where the right slippage is hard to predict
- You want to minimise failed transactions without overpaying on slippage

**Use a fixed `slippageBps` value when:**
- You have specific slippage requirements (e.g. stablecoin swaps where 5-10 bps is sufficient)
- You are building a UI that lets users set their own slippage
- You need deterministic behaviour for testing or simulation

## Edge cases

- **Volatile tokens:** RTSE increases slippage tolerance for tokens with high recent price movement. This reduces transaction failures but means you accept more slippage on volatile pairs.
- **Thin liquidity:** for tokens with very thin pool liquidity, slippage depends heavily on trade size. RTSE accounts for this, but very large trades relative to pool depth may still exceed tolerance.
- **Stablecoins:** RTSE typically assigns very low slippage (5-15 bps) for stablecoin pairs, since their price is predictable. A fixed 10-20 bps is also reasonable for these pairs.
