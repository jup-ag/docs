This doc covers how fee collection and settlement works for enterprise integrators on Swap API v2 under a rev-share agreement. It applies to both the `/order` + `/execute` path (Meta-Aggregator) and the `/build` path (Router).

In both cases the integrator passes the same parameter, `integratorFeeBps`, representing the total fee the end user pays. Jupiter collects the fee into a Jupiter-owned collector wallet, converts the collected mints to USDC on a rolling basis, and settles the integrator's share to a payout wallet once a month.

`integratorFeeBps` is an enterprise-only parameter. It will not work unless your organisation ID has been whitelisted by the Jupiter team for the fee settlement service. It is not part of the public API reference.

## The only fee parameter to use

Swap API v2 exposes different fee parameters for non-enterprise integrators on each path: `referralAccount` and `referralFee` on `/order`, and `platformFeeBps` on `/build`.

Enterprise integrators on the fee settlement service use `integratorFeeBps` only, on both paths. Do not use `referralAccount`, `referralFee`, or `platformFeeBps`.

## What we need from the integrator

For fee settlement, there are two things:

- **Provide the USDC payout wallet.** This is where the integrator's share of fees is settled. The USDC ATA on this wallet must be initialised before the first payout.
- **Pass `integratorFeeBps` in the request.** This is the total bps the end user pays on the swap, passed on `/order` or `/build`.

That's it. Jupiter handles the collector wallet, fee collection, conversion to USDC, reconciliation, and payout.

## Fee model

The integrator passes one number, `integratorFeeBps`: the **total** fee the end user pays on the swap. Jupiter collects that whole amount into the Jupiter-owned collector wallet on chain, then splits it per the commercial rev-share agreement: the integrator keeps **y%** and Jupiter keeps **x%** (`y = 100 − x`). The split is reconciled at settlement, not on chain per swap.

The API Gateway applies the agreed rev-share to each request based on the integrator's org, so the integrator only ever sets the total. For a total fee of **F bps**, the integrator receives `y/100 × F` bps and Jupiter receives `x/100 × F` bps.

Because the integrator sets `integratorFeeBps`, they have everything needed to reconstruct the split per swap or in aggregate.

## Passing the fee

The same `integratorFeeBps` parameter is used on both paths.

### Using with `/order` + `/execute`

Pass `integratorFeeBps` on the `/order` request. `/order` returns an assembled transaction, which you sign and submit through `/execute` for managed landing. Because Jupiter assembles the transaction and `/execute` validates it against the original order before landing, fee inclusion is enforced on this path: the fee instruction cannot be stripped before submission.

```
GET https://api.jup.ag/swap/v2/order
```

Reference: [Order & Execute](/swap/order-and-execute)

### Using with `/build`

Pass `integratorFeeBps` on the `/build` request. `/build` returns raw swap instructions, and the integrator assembles, signs, and submits the transaction either through their own RPC or through `/submit`. `/execute` is not used on this path.

On `/build`, Jupiter is not constructing the transaction the user signs the way it does on `/order`, so it cannot enforce fee inclusion at the point the integrator submits the transaction on chain. For enterprise integrators on the fee settlement service, `integratorFeeBps` communicates the total fee the user is to be charged, and including that fee in the submitted transaction is part of the commercial agreement. Jupiter reconciles the fees collected on chain against what is expected. Once a fee lands in the Jupiter-owned collector wallet, Jupiter has custody of the funds and handles the split before the integrator's share is sent to the payout wallet.

```
GET https://api.jup.ag/swap/v2/build
```

Reference: [Build](/swap/build)

## Fee mint priority

On `/order`, the fee mint is selected automatically based on a priority list, so it can be either the input or output mint. It cannot be forced to always be the input. Read `feeMint` from the `/order` response rather than assume which mint the fee is taken on.

This automatic fee mint priority applies to `/order` only. It does not apply to `/build`.

Reference: [Fee mint priority](/swap/order-and-execute#fee-mint-priority)

## Settlement

Fees are collected by Jupiter and settled in USDC. There are two distinct rhythms, worth keeping separate:

- **Conversion (every 6 hours):** the collector wallet receives fees in whatever mint each swap settles in. Jupiter opens and maintains ATAs for every mint that flows in, and converts the collected balances to USDC every 6 hours.
- **Payout (monthly):** the integrator's share of the accumulated USDC is transferred to the payout wallet once a month.

**Fees are denominated in USDC at the point of the periodic conversion, not at the point of the user swap.**

At settlement the accumulated total is split per the agreement: the integrator's share (**y%**) goes to the payout wallet, and Jupiter retains its share (**x%**) in the treasury wallet.

Fees can arrive in any mint, including long-tail assets where Solana liquidity moves quickly. Opening per-mint ATAs and converting balances yourself adds up in cost and operational overhead, so Jupiter handles this for you.

### What the integrator is responsible for

- The payout wallet address that Jupiter sends to
- Initialising the USDC ATA on that wallet before the first payout

<Note>
A self-managed collector wallet, where Jupiter sends collected mints to a wallet the integrator controls instead of converting and paying out monthly, is not yet supported. Let us know if you need it.
</Note>

### Wallets

The collector and payout wallets are configured per API key as metadata within the team:

- **Collector wallet:** where fees are collected (Jupiter-owned)
- **Treasury wallet:** where Jupiter's share is retained (Jupiter-owned)
- **Payout wallet:** where the integrator's final USDC share is settled (integrator-owned)

## Positive slippage

Positive slippage capture is a separate feature with its own whitelisting and mechanics. If you want to opt into it, see the public docs: [Positive Slippage](/swap/positive-slippage).

## Whitelisting

Before `integratorFeeBps` has any effect, Jupiter must whitelist the organisation. To request whitelisting:

1. Find your **organisation ID** under Settings in the [Developer Platform](https://developers.jup.ag/portal).
2. Send it to us.
3. We enable fee settlement for that organisation.

## Rate limits & credits

Rate limits and credits are decided between the teams as per requirements and vary integrator to integrator. They're agreed commercially rather than fixed by this doc.

As a reference point on how rate limits behave: the limit is applied to the whole org, and all teams within the org fall under the same bucket. We can start off with a number you're comfortable with and then scale up as traffic grows.
