The Swap API supports swapping through Squads V5 multisig vaults. The server wraps Jupiter swap instructions inside the Squads `executeTransactionSyncV2` CPI, so you get back a ready-to-sign transaction without needing the Squads SDK client-side.

Two endpoints:

1. **`GET /squads/v5/order`**: pass the vault PDA as `taker`, along with `settingsPda` and `signers`. Returns a wrapped transaction ready for signing.
2. **`POST /squads/v5/execute`**: submit the fully signed transaction. Jupiter validates and executes via Jupiter Beam.

The vault never signs directly. It "signs" via CPI through the Squads program. Multisig members sign the outer transaction, and the Squads program enforces the threshold.

<Note>
Only **Metis routing** is currently supported for multisig swaps. RFQ is not supported yet.
</Note>

## How it works

<Steps>
  <Step title="Request a wrapped order">
    Call `GET /squads/v5/order` with swap parameters, vault as `taker`, `settingsPda`, and `signers`.
  </Step>
  <Step title="Validate signers on-chain">
    The API fetches multisig settings on-chain and validates the provided signers against the member list and threshold.
  </Step>
  <Step title="Fetch swap instructions">
    The API calls Metis for swap instructions.
  </Step>
  <Step title="Wrap in Squads CPI">
    The API wraps swap instructions in Squads `executeTransactionSyncV2` CPI, simulates the transaction to set compute budget, and returns the wrapped transaction.
  </Step>
  <Step title="Collect signatures">
    All signers listed in the request must sign the transaction. Signing order does not matter, as long as all signatures are present.
  </Step>
  <Step title="Execute">
    Submit the fully signed transaction to `POST /squads/v5/execute`. The API validates signatures, checks the transaction has not been tampered with, and sends it to the network.
  </Step>
</Steps>

## Parameters

### GET /squads/v5/order

```
GET https://api.jup.ag/swap/v2/squads/v5/order
```

| Parameter    | Type     | Required | Description                                                                 |
|-------------|----------|----------|-----------------------------------------------------------------------------|
| `inputMint` | `string` | Yes      | Mint address of the token to sell                                           |
| `outputMint`| `string` | Yes      | Mint address of the token to buy                                            |
| `amount`    | `string` | Yes      | Amount in smallest units (e.g. lamports)                                    |
| `taker`     | `string` | Yes      | Vault PDA (the account holding the tokens being swapped)                    |
| `settingsPda`| `string` | Yes      | Squads V5 settings account address                                          |
| `signers`   | `string` | Yes      | Comma-separated signer pubkeys. **First signer = fee payer.** Order matters |
| `useWsol`   | `string` | No       | `true` to use wrapped SOL. Defaults to `false`                              |

#### Signers

The `signers` parameter is explicitly provided by the caller. The API does not infer who will sign from the settings account.

- Must provide at least `threshold` number of signers (threshold is read from the settings account on-chain)
- Every signer must be a member of the multisig (validated against the on-chain settings)
- **The first signer is the fee payer** and pays transaction fees and rent

#### Response

The response follows the standard order response shape:

- `transaction`: base64-encoded V0 transaction, already wrapped in a Squads `executeTransactionSyncV2` CPI call
- `requestId`: UUID to pass to `/execute`
- `inAmount`, `outAmount`: swap amounts
- `routePlan`: array of swap steps with AMM details

### POST /squads/v5/execute

```
POST https://api.jup.ag/swap/v2/squads/v5/execute
```

| Field                | Type     | Description                              |
|---------------------|----------|------------------------------------------|
| `requestId`         | `string` | UUID from the `/order` response          |
| `signedTransaction` | `string` | Base64-encoded fully signed transaction  |

The execute endpoint does **strict bit-for-bit validation** against the cached transaction from `/order`. No modifications are allowed: you cannot add instructions, change accounts, or alter data. The only difference between the cached transaction and what you submit should be the signatures.

The transaction must be **fully signed** by every signer listed in the `/order` request. Partial signatures are rejected.

## Code examples

### Multi-signer (threshold >= 2)

```typescript
const BASE_URL = "https://api.jup.ag/swap/v2";

// 1. Get the wrapped transaction
const orderRes = await fetch(
  `${BASE_URL}/squads/v5/order?` +
    new URLSearchParams({
      inputMint: "So11111111111111111111111111111111111111112",
      outputMint: "EPjFWdd5AufqSSqeM2qN1xzybapC8G4wEGGkZwyTDt1v",
      amount: "10000000",
      taker: vaultPda.toBase58(),
      settingsPda: settingsPda.toBase58(),
      signers: `${walletPubkey.toBase58()},${deviceKey.publicKey.toBase58()}`,
    })
);
const order = await orderRes.json();

// 2. Deserialise the transaction
const tx = VersionedTransaction.deserialize(
  Buffer.from(order.transaction, "base64")
);

// 3. Sign with all required multisig members
// If threshold >= 2, sign with additional keys first
tx.sign([deviceKey]);

// Then sign with the primary wallet
const signed = await wallet.signTransaction(tx);

// 4. Submit for execution
const execRes = await fetch(
  `${BASE_URL}/squads/v5/execute`,
  {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      requestId: order.requestId,
      signedTransaction: Buffer.from(signed.serialize()).toString("base64"),
    }),
  }
);
const result = await execRes.json();
// result.signature — the transaction signature
```

## Key details

- **Signer order matters.** The first address in `signers` becomes the fee payer. If you put the wrong wallet first, that wallet pays gas.
- **Transaction is immutable.** After receiving the transaction from `/order`, you can only add signatures. Any modification to the transaction bytes will fail validation on `/execute`.
- The vault PDA and settings PDA are derived from the Squads V5 program (`SMRTzfY6DfH5ik3TKiyLFfXexV8uSG3d2UksSCYdunG`).
- Address Lookup Tables from the original swap transaction are reused to keep the wrapped transaction within Solana's 1232-byte limit.
- The vault PDA never appears as a transaction-level signer. It signs through the Squads program CPI.
